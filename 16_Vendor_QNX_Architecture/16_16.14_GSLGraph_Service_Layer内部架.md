## 16.14 GSL(Graph Service Layer)内部架构深度解析

### 16.14.1 概述与架构定位

GSL(Graph Service Layer)是DSP端AudioReach架构的Graph服务层，位于AGM Service和APM(SPF)之间。在SA8295虚拟化架构下，GSL运行在QNX域(PVM)，Android域(GVM)的AGM Service通过gsl_fe→MM-HAB→gsl_vm_be跨VM通道与GSL交互。GSL负责Graph的创建、状态管理、数据路径配置和GPR通信，是DSP Graph操作的核心引擎。

**源码路径**：`Qnx/apps/qnx_ap/AMSS/multimedia/audio/audio_ar/audio_driver/audio_reach/gsl/`（QNX PVM 侧，`src/` 含 15 个 `.c`、`inc/` 含头文件）

> **源码说明**：GSL 运行于 **QNX PVM**，真实源码位于上述 `audio_reach/gsl/` 路径。已核实 `gsl/src/` 恰为 15 个 `.c`：gsl_main / gsl_graph / gsl_datapath / gsl_subgraph / gsl_subgraph_pool / gsl_shmem_mgr / gsl_msg_builder / gsl_common / gsl_hw_rsc_mgr / gsl_dynamic_module_mgr / gsl_mdf_utils / gsl_global_persist_cal / gsl_rtc / gsl_rtc_main / gsl_spf_ss_state。

### 16.14.2 Graph状态机

GSL Graph遵循严格的状态机模型，每个状态对应特定的SPF命令和允许的操作（真实源码枚举 `enum gsl_graph_states`，[`gsl_graph.h`](Qnx/apps/qnx_ap/AMSS/multimedia/audio/audio_ar/audio_driver/audio_reach/gsl/inc/gsl_graph.h:31)）。

> **源码说明**：源码 `enum gsl_graph_states` 实际只有 **7 个状态**：`GRAPH_IDLE / GRAPH_OPENED / GRAPH_STARTED / GRAPH_STOPPED / GRAPH_CLOSED / GRAPH_ERROR_ALLOW_CLEANUP / GRAPH_ERROR`。**并无独立的 PREPARED 状态**——`prepare` 是在 OPENED/STOPPED 态下执行的操作，不改变 `graph_state` 枚举值。下图按真实枚举绘制。

```mermaid
stateDiagram-v2
    IDLE --> OPENED : gsl_graph_open<br/>APM_CMD_GRAPH_OPEN
    OPENED --> STARTED : gsl_graph_prepare+start<br/>APM_CMD_GRAPH_START
    STARTED --> STOPPED : gsl_graph_stop<br/>APM_CMD_GRAPH_STOP
    STOPPED --> STARTED : gsl_graph_prepare+start
    STOPPED --> CLOSED : gsl_graph_close<br/>APM_CMD_GRAPH_CLOSE
    OPENED --> CLOSED : gsl_graph_close<br/>APM_CMD_GRAPH_CLOSE
    CLOSED --> IDLE : 去初始化
    STARTED --> ERROR_ALLOW_CLEANUP : 非致命错误<br/>允许close/stop
    ERROR_ALLOW_CLEANUP --> CLOSED : gsl_graph_close
    ERROR_ALLOW_CLEANUP --> ERROR : 致命错误升级
    ERROR : 不可恢复<br/>无SPF操作允许
```

**状态详细说明**：

| 状态 | 描述 | 允许的操作 |
|------|------|-----------|
| GRAPH_IDLE | 初始化完成，等待open | open |
| GRAPH_OPENED | Graph在SPF上成功打开 | prepare, close, set_config, add_new, change |
| GRAPH_STARTED | Graph在SPF上运行 | stop, write, read, set_config, flush |
| GRAPH_STOPPED | Graph在SPF上停止 | prepare, close, start |
| GRAPH_CLOSED | Graph在SPF上关闭，未去初始化 | re-init |
| GRAPH_ERROR_ALLOW_CLEANUP | 非致命错误，允许清理操作 | close, stop |
| GRAPH_ERROR | 不可恢复错误 | 无SPF操作 |

**控制信号分组**：

| 信号组 | 包含操作 | 说明 |
|--------|---------|------|
| GRAPH_CTRL_GRP1_CMD_SIG | OPEN, CLOSE, PREPARE, START, STOP, CONFIGURE_READ/WRITE_BUFFERS | 串行化控制命令 |
| GRAPH_CTRL_GRP2_CMD_SIG | SET_CFG, REGISTER_CFG, FLUSH, REGISTER_MODULE_EVENTS | 并行配置命令 |

#### 16.14.7.1 GPR：Generic Packet Router（GSL↔SPF 的报文路由）

> **源码验证**：`audio_reach/gpr/`，含 `api / core / datalinks / platform / ext`。域ID 定义见 `gpr/api/gpr_ids_domains.h`，包结构见 `gpr/api/gpr_packet.h`。

GSL 本身**不直接**与 ADSP 通信，而是把每一条 SPF 命令封装成 **GPR 报文**，交给 GPR 路由到目标域。GPR 是 AudioReach 栈里"谁给谁发消息"的统一寻址与投递层。

**GPR 域 ID（`gpr_ids_domains.h`，报文的收发方寻址）**：

| 域 ID | 宏 | 含义 |
|-------|----|------|
| 1 | `GPR_IDS_DOMAIN_ID_MODEM_V` | Modem 子系统 |
| 2 | `GPR_IDS_DOMAIN_ID_ADSP_V` | **ADSP（SPF/APM 所在，音频主战场）** |
| 3 | `GPR_IDS_DOMAIN_ID_APPS_V` | Apps（AP 侧，QNX 上的 GSL 即挂此域） |
| 4 | `GPR_IDS_DOMAIN_ID_SDSP_V` | Sensor DSP |
| 5 | `GPR_IDS_DOMAIN_ID_CDSP_V` | Compute DSP |

一条 GPR 报文头（`gpr_packet_t`，见 `gpr_packet.h`）通过位域打包携带：`version`、`header_size`、`packet_size`、**`src_domain_id`/`dst_domain_id`**（源/目标域）、`src_port`/`dst_port`（模块实例 IID）、`token`（异步应答配对）、`opcode`（APM/模块命令码）。GSL 发命令时填 `src=APPS / dst=ADSP`，ADSP 回应答/事件时填 `src=ADSP / dst=APPS`，GPR 据此双向投递。

**GPR 源码组成**：

| 目录 | 职责 |
|------|------|
| `gpr/core/src/` | 路由核心：`gpr_drv.c`（驱动/注册/路由）、`gpr_main.c`（初始化）、`gpr_memq.c`（报文内存队列/池）、`gpr_list.c`（链表）；均有 `*_island.c` 变体供低功耗 island 模式使用 |
| `gpr/datalinks/gpr_glink` | 数据链路：SA8295 上 AP↔ADSP 走 **GLink**（共享内存 IPC），GPR 把报文交给 GLink 物理传输 |
| `gpr/platform/qnx` | QNX 平台适配（线程/中断/内存与 QNX 对接） |
| `gpr/api/` | 对外头文件：`gpr_api.h`/`gpr_api_inline.h`（发送/注册回调）、`gpr_packet.h`（包结构）、`gpr_ids_domains.h`（域）、`ipc_dl_api.h`（数据链路抽象） |

> **一句话定位**：GSL 用 `gsl_msg_builder.c` 拼好 payload → 交 GPR（`gpr_drv.c`）按 `dst_domain_id=ADSP` 路由 → GLink 送达 ADSP 的 APM → 应答带 `token` 原路返回，`gsl_common.c` 凭 token/信号唤醒等待的调用线程。

#### 16.14.7.2 ar_osal：跨 OS 抽象层（QNX 与 Linux 复用同一套 GSL 的关键）

> **源码验证**：`audio_reach/ar_osal/`，`api/` 下 17 个抽象头，`src/` 下**只有 `qnx/` 一个实现目录**（13 个 `.c`）。

AudioReach 栈能同时跑在 QNX(SA8295 PVM) 与 Linux（其他平台）上，靠的就是 ar_osal：**GSL/GPR/ACDB 只调用 `ar_osal_*` 抽象接口，绝不直接调 QNX 原生 API**。移植到新 OS 时，只需在 `ar_osal/src/<os>/` 补一套实现，上层零改动。

| 抽象头 | 屏蔽的 OS 原语 |
|--------|---------------|
| `ar_osal_mutex.h` | 互斥锁：`ar_osal_mutex_create/lock/try_lock/unlock/destroy` |
| `ar_osal_signal.h` / `ar_osal_signal2.h` | 信号/事件：`signal_set/clear/wait/timedwait`（GSL 等 SPF 应答的超时等待即靠它） |
| `ar_osal_thread.h` | 线程创建/join/优先级 |
| `ar_osal_sleep.h` / `ar_osal_timer.h` | 延时与定时 |
| `ar_osal_heap.h` / `ar_osal_mem_op.h` | 堆分配与内存操作 |
| `ar_osal_shmem.h` | 共享内存（GSL OOB Buffer 底层依赖） |
| `ar_osal_file_io.h` | 文件 IO（ACDB 读 `.acdb`/`.qwsp` 文件） |
| `ar_osal_string.h` / `ar_osal_log.h` / `ar_osal_log_pkt_op.h` | 字符串/日志/日志打包 |
| `ar_osal_servreg.h` / `ar_osal_sys_id.h` | 服务注册（SSR 感知）/子系统 ID |

```mermaid
graph TB
    GSL["gsl / gpr / acdb 上层"] -->|"仅调用 ar_osal_* 接口"| API["ar_osal/api<br/>17个抽象头"]
    API --> QNXIMPL["ar_osal/src/qnx<br/>QNX原生实现"]
```

> **对 SA8295 的意义**：QNX 侧 `gsl/src` 那 15 个 `.c` 与 Android 侧的 GSL 是同源代码，差异只体现在 `ar_osal/src/qnx` 这一层薄适配。这解释了为何"同一套 AudioReach 能既服务 QNX PVM、又服务 Linux/Android"。

### 16.14.8 gsl_FE与gsl_BE：GSL虚拟化前后端

在SA8295虚拟化架构中，GSL（Graph Service Layer）被分为**VM前端(gsl_fe)**和**VM后端(gsl_vm_be)**，运行在不同的VM域中，通过MM-HAB跨VM通道通信。这是QNX主控架构在GSL层的核心实现机制。

> **源码验证**：本节内容基于`code/qc`目录下的实际源码验证：
> - QNX侧gsl_vm_be：`Qnx/apps/qnx_ap/AMSS/multimedia/audio/audio_ar/audio_driver/gsl_be/`
> - Android侧gsl_fe：`Android/lagvm/.../prebuilt_HY11/.../gsl_fe/`（预编译库libar-gsl_fe.so）
> - 共享消息定义：`gsl_vm_msg.h` / `gsl_vm_common.h`

#### VM前后端架构定义

| 概念 | 全称 | 运行位置 | 说明 |
|------|------|---------|------|
| gsl_fe | GSL VM Frontend | Android域(GVM) | 预编译库libar-gsl_fe.so，将GSL API调用序列化为VM Opcode，通过MM-HAB发送给QNX侧 |
| gsl_vm_be | GSL VM Backend | QNX域(PVM) | 接收HAB消息，反序列化Opcode，调用QNX本地GSL API（gsl_open/gsl_read/gsl_write等），直接与ADSP交互 |
| MM-HAB | Multimedia Hypervisor Audio Bus | 跨VM通道 | Qualcomm的跨VM通信机制，基于habmm_socket API，音频通道：tx=MM_AUD_1, rx=MM_AUD_2 |

> **关键区别**：gsl_fe/gsl_vm_be是**虚拟化层级**的FE/BE概念，不是DSP内部AFE/ABE概念。gsl_fe在Android域代理所有GSL API调用，gsl_vm_be在QNX域执行真正的GSL操作。Android域完全不直接访问ADSP，所有GSL请求必须通过gsl_fe→MM-HAB→gsl_vm_be路径。

#### MM-HAB通信机制

gsl_vm_be在QNX侧通过MM-HAB与Android域通信，核心代码流程（源码：gsl_be.c）：

1. **BE服务启动**：gsl_vm_be服务在QNX侧启动，通过`habmm_socket_open()`打开两个HAB通道
   - `MM_AUD_1`：命令通道(tx)，接收Android域的GSL请求
   - `MM_AUD_2`：事件通道(rx)，向Android域发送DSP事件通知
   - 标志：`HABMM_SOCKET_OPEN_FLAGS_SINGLE_BE_SINGLE_FE`（一对一连接）

2. **请求处理循环**：每个GVM会话一个线程
   - `habmm_socket_recv(tx_hab_handle)` → 阻塞等待Android域请求
   - 解析Opcode → 调用对应的QNX本地GSL API（如`gsl_open()`、`gsl_read()`）
   - `habmm_socket_send(tx_hab_handle)` → 发送响应回Android域

3. **事件回调通道**：ADSP事件从QNX侧推送到Android域
   - GSL事件回调 → `habmm_socket_send(rx_hab_handle)` → Android域gsl_fe接收

4. **会话管理**：最多支持8个GVM会话（`GSL_BE_MAX_SESSIONS = 8`），每个会话独立线程

#### 双内存模式

| 模式 | 值 | 说明 | OOB缓冲区操作 |
|------|-----|------|-------------|
| GSL_VM_MEM_TYPE_HOST | 0 | PVM(QNX)侧分配内存 | GVM用REQUEST_BUFF/RELEASE_BUFF |
| GSL_VM_MEM_TYPE_GUEST | 1 | GVM(Android)侧分配内存 | GVM用PROVIDE_BUFF/RETRIEVE_BUFF |

> V2版本(GSL_VM_VERSION_2)支持GUEST内存模式，允许Android域分配共享内存，减少一次HAB数据拷贝。

#### 典型场景的VM前后端交互流程

| 音频场景 | Android域(gsl_fe) | MM-HAB | QNX域(gsl_vm_be) | ADSP |
|---------|-------------------|--------|-------------------|------|
| 媒体播放 | PAL→gsl_fe发送OPEN+WRITE | OPCODE_OPEN→OPCODE_WRITE | gsl_open()→gsl_write() | APM创建Graph→处理音频 |
| 麦克风录音 | PAL→gsl_fe发送OPEN+READ | OPCODE_OPEN→OPCODE_READ | gsl_open()→gsl_read() | APM创建Graph→回传音频 |
| 校准设置 | acdb-loader→gsl_fe发送SET_CAL | OPCODE_SET_CAL | gsl_set_cal() | APM应用校准 |
| 安全音频 | (不经gsl_fe) | (不经MM-HAB) | QNX直通GSL | APM处理安全流 |
| ADSP事件 | gsl_fe接收事件回调 | Event via rx_hab | 事件回调触发send | APM→GSL→BE→FE |

### 16.14.9 端到端全链路：一次播放从 PAL 到 ADSP 的跨 VM 时序

把前面各层串起来，下图展示 GVM(Android) 上一次媒体播放请求，如何穿过 PAL→AGM→gsl_fe→MM-HAB→gsl_be→本地 GSL→GPR→ADSP，以及缓冲区数据如何回流。这是理解"QNX 主控、Android 借道"的整张全景图。

```mermaid
sequenceDiagram
    participant APP as App/AudioFlinger(GVM)
    participant PAL as PAL(GVM)
    participant AGM as AGM(GVM)
    participant FE as gsl_fe(GVM)
    participant HAB as MM-HAB(habmm)
    participant BE as gsl_vm_be(PVM)
    participant GSL as 本地GSL gsl_graph(PVM)
    participant GPR as GPR gpr_drv(PVM)
    participant ADSP as APM/SPF(ADSP)

    APP->>PAL: openOutputStream + write
    PAL->>AGM: 打开会话/设置GKV
    AGM->>FE: gsl_open/gsl_write(GSL API)
    FE->>HAB: 序列化为VM Opcode(OPEN/WRITE)
    HAB->>BE: habmm_socket_recv(tx=MM_AUD_1)
    BE->>GSL: 反序列化 调本地gsl_open/gsl_write
    GSL->>GPR: gsl_msg_builder拼包 dst=ADSP
    GPR->>ADSP: GLink投递GPR报文(APM命令)
    ADSP->>ADSP: 创建/运行Graph 处理音频
    ADSP-->>GPR: 应答/事件(带token)
    GPR-->>GSL: gsl_common按token+信号唤醒
    GSL-->>BE: 返回结果
    BE-->>HAB: 序列化应答(rx=MM_AUD_2)
    HAB-->>FE: habmm_socket_recv
    FE-->>AGM: 返回GSL调用结果
    AGM-->>PAL: 会话就绪/写入完成
    PAL-->>APP: 播放中
```

**读图要点**：
- **控制面**（OPEN/START/校准）与**数据面**（WRITE/READ 的 PCM 缓冲）走同一条 MM-HAB 通道，但数据面 Buffer 采用 OOB 共享内存（`gsl_shmem_mgr.c`），HAB 只传描述符不搬大数据。
- `token` 贯穿始终：从 gsl_fe 发起到 ADSP 应答原路返回，是跨 4 层（FE/BE/GSL/GPR）异步配对的唯一线索。
- **安全音频/QNX 本地音频**不经 gsl_fe/HAB，PVM 内直接 `本地GSL→GPR→ADSP`，省去跨 VM 开销（见 16.14.8 场景表"安全音频"行）。
